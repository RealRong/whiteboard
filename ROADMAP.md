# Whiteboard 低代码路线图（工业互联网）

## 目标
- 将白板作为低代码二开基础。
- 让初级开发无需理解渲染与算法细节，也能交付行业应用。
- 以 Web 轻量化替代 MFC/Qt 系统。

## 指导原则
- 核心逻辑放在 `whiteboard-core`，渲染与交互放在 `whiteboard-react`，业务扩展放在 `whiteboard-plugins`。
- 配置优先、插件驱动、事件化集成。
- 数据模型以“最优设计”为前提，无历史兼容包袱。
- 面向大图的性能与稳定性。

## 阶段规划

### Phase 0（0-2 周）- 基础能力
**目标：** 稳定的核心类型 + Edge 系统 + 扩展槽位。
- Edge 基线能力：路由类型、控制点、箭头/标记、dash/animated、命中测试。
- Node/Edge 注册表：以声明式方式注册类型。
- 扩展点：工具栏、右键菜单、属性面板、命令系统、导入导出钩子。
- 事件总线：选中、编辑、连接、校验、数据变更的生命周期事件。
- Demo 验证：基础节点 + 边 + 端点重连交互体验。

**交付物**
- `whiteboard-core`：edge 路由、自动锚点、state/commands。
- `whiteboard-react`：edge layer/handles、注册表 hooks。
- `apps/whiteboard-demo`：edge 行为展示。

### Phase 1（1-3 个月）- 低代码可用
**目标：** 让初级开发依靠配置与模板快速交付。
- Schema 驱动的 node/edge 定义（label、端口、数据字段、校验规则）。
- 属性面板构建器：JSON schema -> 表单 UI。
- 轻量规则引擎：样式随数据变化、条件显示/隐藏、提示规则。
- 模板库：工业节点（设备、阀门、传感器、PLC、控制柜）。
- 版本化的导入导出。

**交付物**
- `whiteboard-core`：schema 校验、规则运行时。
- `whiteboard-react`：属性面板渲染器 + 动态控件。
- `whiteboard-plugins`：工业符号包 v1。

### Phase 2（3-6 个月）- 工业可用
**目标：** 满足工业互联网的实际落地要求。
- 实时数据绑定：节点/边绑定点位并驱动样式更新。
- 告警可视化：边/节点闪烁、标记、叠加提示。
- 图层与底图：厂区平面图/工艺图叠加。
- 权限与审计：角色权限、操作日志。
- 性能扩展：虚拟化渲染、空间索引、增量重绘。

**交付物**
- `whiteboard-core`：数据绑定 API、告警状态模型。
- `whiteboard-react`：图层管理器、告警效果、渲染优化。
- `whiteboard-plugins`：告警面板 + 点位适配器。

### Phase 3（6-9 个月）- 平台化
**目标：** 集成能力和生态建设。
- 嵌入式 SDK（iframe/微前端）+ postMessage API。
- 插件市场结构与打包工具。
- 多图管理、搜索与复用库。
- 协作（可选）：多用户编辑、锁定或 OT。

**交付物**
- SDK 文档 + 集成 Demo。
- 插件打包工具 + 模板仓库。

## 优先级清单
**必须有（低代码基础）：**
- 注册表 + 插件系统
- Schema 驱动 node/edge
- 属性面板构建器
- 基础规则引擎
- 版本化导入导出

**可加分（工业差异化）：**
- 实时数据绑定
- 告警可视化
- 权限/审计
- 底图/多图层

## 推荐拆分
- `whiteboard-core`：数据模型、算法、路由、校验、规则运行时。
- `whiteboard-react`：渲染、交互、面板、编辑器 UI。
- `whiteboard-plugins`：工业符号库、告警面板、行业适配。

## 风险与应对
- 复杂度膨胀：严格插件边界，避免跨层耦合。
- 性能回退：尽早引入虚拟化与限制策略。
- 需求膨胀：每阶段先锁定 MVP，避免可选功能提前进入。

## 近期下一步
- 确认 edge 体验细节（自动锚点、重连视觉）。
- 完成注册表接口与 schema 格式定义。
- 明确属性面板渲染器的契约。
